## 1. プロダクト概要

### コアビジョン（最終形）
> 「あなたの麻雀の歴史」を蓄積し、上達を可視化する。

**売るもの**: 対局体験でも解説でもなく、**「自分が上達している実感」**。
**競合優位の源泉**: ユーザーのデータが蓄積されるほど価値が上がる **データの堀（モート）**。
**作らないもの**: 対局エンジン・リアルタイム対戦・AI対戦。

### 進化ロードマップ（戦略の核心）

```
v0.1: AI手牌解説          ← 集客フック。競合は存在するが最速で検証できる
  ↓
v0.2: 解説 + 記録機能      ← データ蓄積開始。ここから独占が始まる
  ↓
v1.0: 上達ジャーニー可視化 ← 本来の製品。「あなたの麻雀の歴史」が堀になる
  ↓
v1.x: グループ・コミュニティ ← ネットワーク効果で強固化
```

**なぜこの順序か**: v0.1のユーザーが持つデータがv1.0の価値を決める。最初から「データを蓄積する器」として設計することで、競合がv0.1の解説機能をコピーした日には、ユーザーはすでに数ヶ月分のデータをこのアプリに預けている。

### 差別化ポイント
既存アプリは「解析して終わり」。このアプリは「**継続記録が価値になる**」設計。
NAGAは評価スコアを出す。このアプリは「**なぜそのミスが繰り返されるか**」を可視化する。

## 2. ターゲットユーザー

**メインターゲット**: 麻雀中級者（雀歴1〜3年、段位戦で伸び悩んでいる層）

**課題の一文**:
「麻雀アプリで毎日打っているのに、同じミスを繰り返している。なぜ自分の選択が悪かったのかを教えてくれる人も場所もない。」

**初級者を外す理由**: ルール習得が先でAI解説の価値が伝わらない。
**上級者を外す理由**: 既に天鳳牌譜解析ツールや上級者コミュニティを持っている。

## 3. バージョン別スコープ

### v0.1 MVP ── 集客フック（〜3ヶ月）
**目標**: AI解説の価値を検証し、最初のユーザーを獲得する

**IN（作るもの）**
- 手牌14枚をテキスト入力（`1m2m3m4p5p6p7s8s9s東南西北` 形式）
- AIが捨て牌候補TOP3 ＋ 各理由（向聴数・打点・守備）を返す
- 結果のX（Twitter）シェアボタン
- 無料枠（月10回）と有料プラン（月額980円・無制限）の境界

**OUT（作らないもの）**
| 機能 | 理由 |
|---|---|
| ユーザー認証・ログイン | v0.2で追加 |
| 解析履歴の保存 | v0.2で追加 |
| 牌ボタンUI | フィードバック次第でv0.2に追加 |
| 河・点数状況の入力 | v0.2で追加 |
| 対局エンジン・リアルタイム対戦 | 対象外 |

**成功条件**: 課金者10人・Week4フィードバック取得

---

### v0.2 ── データ蓄積開始（3〜6ヶ月）
**目標**: 「記録が溜まるほど価値が上がる」設計に移行する

**追加機能**
- **ユーザー認証**（Supabase Auth）← ここがターニングポイント
- 解析ごとに「この判断を記録しますか？」を表示 → DynamoDB/Supabase に保存
- 解析履歴の一覧表示（直近20件）
- 局面コンテキスト入力（河・点数状況・リーチ有無・東南場）
- 牌ボタンUI（v0.1フィードバックで「入力が面倒」が多ければ）

**成功条件**: 1ヶ月以上継続利用ユーザーが課金者の50%以上

---

### v1.0 ── 上達ジャーニー（6〜12ヶ月）
**目標**: 「あなたの麻雀の歴史」を価値にする。競合が解説機能をコピーしても勝てる状態。

**追加機能**
- **上達グラフ**: 時系列での判断精度スコアの推移
- **弱点レポート**: 「あなたは〇〇局面での判断ミスが多い」をAIが月次で提示
- **対局記録との紐付け**: 実際の対局結果（勝敗・順位）と解析データを連携
- Mortal + Claude 統合エンジン（精度向上）
- 牌譜インポート（天鳳/雀魂URL）← 入力摩擦をゼロにする

**成功条件**: MRR 5万円・月次解約率10%未満

---

### v1.x ── ネットワーク効果（12ヶ月以降）
- グループ機能（仲間内でのランキング・成長比較）
- コミュニティフィード（面白い局面をシェア・コメント）
- 雀荘向けSaaS展開（対局記録の業務ツール化）

## 4. 技術スタック

### v0.1 構成（最小）
```
フロントエンド : Next.js 14（App Router）+ shadcn/ui (Tailwind)
デプロイ       : Vercel（git push で自動デプロイ）

バックエンド   : FastAPI (Python 3.12) on AWS Lambda (SAM)
               ※ Python/AWS の既存スキルを直接活用

AI            : Anthropic Claude API（claude-3-5-sonnet）
DB            : なし（認証・履歴なし）
```

### v0.2 追加構成（認証・DB）
```
認証          : Supabase Auth（Google/メール認証）
               ※ Firebase より設定が簡単・PostgreSQL が使える
DB            : Supabase PostgreSQL（解析履歴・ユーザー情報）
               ※ DynamoDB より SQL で扱いやすく、社長スキル（RDS）と近い
```

### v1.0 追加構成（Mortal統合）
```
Mortal Engine : サードパーティMortal API → 将来的にLambdaコンテナへ自己ホスト
```

### v0.1 アーキテクチャ
```
ユーザー（ブラウザ / スマホ）
    ↓ 手牌入力
Next.js on Vercel
    ↓ API call
API Gateway（AWS）
    ↓
Lambda（FastAPI + Mangum）
    ↓
Claude API（Anthropic）
    ↓ JSON レスポンス
ブラウザに解説表示
```

### v0.2 アーキテクチャ（追加分）
```
Next.js
    ↓ 認証
Supabase Auth
    ↓ 解析結果の保存
Supabase PostgreSQL
    ↓ 履歴・分析データ
Next.js（解析履歴ページ）
```

## 4-B. Phase 2 技術設計: Mortal + Claude 統合エンジン

### 設計思想
```
Mortal  : 「何を切るべきか」を統計的に算出（数千万局学習の専用AIの精度）
Claude  : 「なぜそれを切るべきか」を中級者向けに自然言語で解説

→ 精度は Mortal が保証し、理解しやすさは Claude が担う
```

### アーキテクチャ
```
ユーザー入力（手牌 + 局面コンテキスト）
        ↓
  ┌─────────────────┐
  │  Mortal Engine  │  ← Lambda コンテナ or サードパーティAPI
  │  （専用麻雀AI）  │
  └────────┬────────┘
           ↓ 評価結果（JSON）
  {
    "recommended": "7s",
    "confidence": 0.87,
    "evaluation_diff": +0.34,
    "alternatives": [
      {"tile": "6s", "eval": -0.12, "note": "受け入れ広いが打点落ちる"},
      {"tile": "東",  "eval": -0.28, "note": "安全牌確保優先の選択"}
    ],
    "shanten": 1,
    "winning_tiles": ["4s","7s","1m","4m"]
  }
        ↓
  ┌─────────────────┐
  │   Claude API    │  ← Mortal出力をコンテキストとして受け取る
  │  （解説生成）   │
  └────────┬────────┘
           ↓ 自然言語解説（JSON）
  {
    "main_explanation": "Mortalは7sを切ることを推奨しています（確信度87%）。
                         現在の手牌は1向聴で、7sを切ることで4s・7s・1m・4mの
                         4種類に待てる形に移行します...",
    "why_not_alternatives": "6sを切る選択肢もありますが、タンヤオが崩れるため
                              打点が下がります...",
    "key_lesson": "今局面で最も大切なこと: 受け入れ枚数と打点のバランス"
  }
```

### Mortal の調達方法（3案）

| 方法 | コスト | 難易度 | 推奨度 |
|---|---|---|---|
| **A: サードパーティAPI利用** | API従量課金（数円/回） | ★☆☆ | ◎ Phase2初期 |
| **B: Docker コンテナで自己ホスト** | Lambda Container（月数百円） | ★★☆ | ○ スケール後 |
| **C: Mortal をソースからビルド** | ゼロ（OSS） | ★★★ | △ 将来の選択肢 |

**Phase2初期は方法Aを推奨**。Mortal互換のAPIサービスが存在し、実装コストを最小化できる。

## 5. 受け入れ基準

**AC-01: 基本動作**
- Given: 手牌14枚をテキスト入力している
- When: 「解説する」ボタンを押す
- Then: 30秒以内に捨て牌候補TOP3と各理由（向聴数・打点・守備）が表示される

**AC-02: 解説の質**
- Given: 麻雀中級者（雀歴1〜3年）が解説画面を初めて見ている
- When: AI解説を読む
- Then: 「なぜその選択が良い/悪いか」の根拠が具体的な数値・役名付きで分かる（初回5人のユーザーインタビューで「意味が分かった」評価4/5以上）

**AC-03: 課金導線**
- Given: 無料ユーザーが月10回の解析を使い切った
- When: 11回目を試みる
- Then: 月額プランへの案内と「翌月まで待てる」選択肢が両方表示される

**AC-04: 課金後の継続体験**
- Given: 月額プランに課金している
- When: 解析を実行する
- Then: 無制限に解析でき、次回更新日と解析履歴（直近20件）が確認できる

**AC-05: モバイル表示**
- Given: スマートフォン（iPhone Safari / Android Chrome）でアクセスしている
- When: トップページを開く
- Then: 手牌入力フォームと解説結果が画面幅に収まり、操作可能な状態で表示される
