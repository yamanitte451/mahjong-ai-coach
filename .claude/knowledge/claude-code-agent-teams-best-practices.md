## 1. アーキテクチャ設計原則

### brain-project分離（記事1）
Why/What（戦略・要件）とHow（実装コード）を明確に分離する。

| 役割 | 本リポジトリでの対応ファイル |
|---|---|
| 脳（Why/What） | `investment_agent_strategy.md`, `requirements.md`, `business_context.md` |
| 実装（How） | `src/` 配下のコード、`infra/` 配下のBicep |
| 知識（Tech Notes） | `.claude/knowledge/` 配下（本ファイル） |

### 3層自律アーキテクチャ（記事1）
全機能を一度に自動化せず、段階的にAIへ委譲する。

| 層 | 本リポジトリでの例 | Sprint |
|---|---|---|
| 実行層（人間確認なしに実行） | Backend/Infra がコード生成 | Sprint 1〜 |
| 判断層（複数候補から選択） | CIO が買い/売り/待ちを判断 | Sprint 2〜 |
| 常時稼働層（自律監視・行動） | Azure Functions の定期監視 | Sprint 3〜 |

**示唆**: 「AIの自律性は全か無かではなく、レイヤーごとに段階的に委譲するのが現実的」
→ Sprint 1でルールベース監視を確立してから、Sprint 2以降でLLM判断を追加する方針の妥当性を裏付ける。

## 2. ファイル構成の役割分担

### AGENTS.md vs CLAUDE.md（記事2）
ツール断片化を防ぐため用途で使い分ける。

| ファイル | 対象ツール | 主な内容 |
|---|---|---|
| `AGENTS.md` | 全AIツール共通（Copilot/Cursor/Claude） | コーディング規約・コマンド・Gitフロー・行動境界 |
| `CLAUDE.md` | Claude Code専用 | エージェントチーム構成・スポーンルール・CoS行動原則 |
| `.claude/skills/` | Claude Code専用 | 再利用可能な手順書（投資判断・開発フロー等） |

### 最小ハーネス設計 3要素（記事3）
複雑なフレームワーク不要。この3要素で十分制御できる。

| 要素 | ファイル | 役割 |
|---|---|---|
| Hook | `~/.claude/settings.json` | プロンプト送信時に自動発火・リマインド |
| CLAUDE.md | `CLAUDE.md` | 行動制約の憲法（何をすべきか・何をしてはいけないか） |
| Skills | `.claude/skills/` | 再利用可能な手順書（どうやるか） |

**重要**: ドキュメントは「人間が読む」ではなく「Claudeが読んで即行動できる」形式で書く。

## 3. Agent Teams 設計の勘所

### モデル使い分け戦略（記事4）
全エージェントを同一モデルにするとコストが高騰する。

| 戦略 | 対象エージェント | モデル | 用途 |
|---|---|---|---|
| deep（高精度） | CoS・CIO・CSO・CFO・CLO | claude-opus-4-6 | 統合判断・戦略立案 |
| adaptive（バランス） | PM・Backend・Infra | claude-sonnet-4-6 | 実装・設計 |
| fast（高速） | Quant・RiskManager | claude-haiku-4-5 | 数値判定・ルール評価 |
| budget（低コスト） | Researcher・QA | claude-haiku-4-5 | 情報収集・テスト生成 |

**コスト感**:
- claude-sonnet-4-6: 入力 $3/MTok、出力 $15/MTok
- claude-haiku-4-5: 入力 $0.25/MTok、出力 $1.25/MTok
- 月間推定（10銘柄監視）: $1〜5程度

### フラット構造を維持する（記事4）
CIOがQuantを内部スポーンするようなネスト構造はコスト高。**CoSが全スポーンを管理するフラット構造**が最適（本リポジトリの現設計は適切）。

### 自動化の限界（記事4）
| 機能 | 実態 | 対策 |
|---|---|---|
| スキル自動注入 | 実質リマインダー、忘れることがある | 重要手順はCLAUDE.md本体にも記載 |
| ドメイン自動検出 | Claudeの自然言語理解で代替可能 | CoSのルーティングルールを具体的に書く |
| Teammate内Subagent | トークンコストに見合わない | CoSがフラットに管理 |

## 6. Agent Teams 自律改善ロードマップ

Agent Teamsが人間の介入なしに自己改善を進めていくための段階的な設計方針。

### Phase 1: 観察・記録の自動化（Sprint 3完了後〜）

各エージェントが自分の判断結果・精度・問題点を自動記録する。

| タスク | 担当 | 実装方法 |
|---|---|---|
| シグナル誤検知の自動記録 | RiskManager | AlertHistory に `false_positive` フラグを追加 |
| スコアリング精度のトラッキング | Quant | スクリーニング結果と実際の株価動向を Cosmos DB に蓄積 |
| 通知の開封率・反応時間の記録 | CoS | Discord Webhook のレスポンスをログ化 |
| 投資判断の事後検証ログ | CIO | 買い推奨後の株価推移を定期記録 |

### Phase 2: パターン分析・自己診断（Sprint 4〜）

蓄積データをもとにエージェントが自分の弱点を発見する。

| タスク | 担当 | 実装方法 |
|---|---|---|
| 誤検知パターンの分析 | Quant + RiskManager | 月次バッチで False Positive 率を集計・CLAUDE.md に反映提案 |
| スコアリング閾値の最適化 | Quant | 過去3ヶ月のスクリーニング結果から閾値を再提案 |
| シグナル優先度の見直し | CIO | SELL-01〜11 の発動頻度・有効率を分析して優先度を再評価 |
| CoS ルーティング精度の検証 | CoS | 誤ルーティングのログを蓄積し、CLAUDE.md の振り分けルールを改訂 |

### Phase 3: CLAUDE.md の自律更新（長期目標）

エージェントがロール定義を自分でアップデートできる仕組み。

```
自律改善サイクル（月次）:
1. 各エージェントが「自分への改善提案」をCoSに送信
2. CoSが提案を統合・優先度付け
3. 社長に変更提案をまとめてレポート
4. 社長の承認後、PM が agents/*/CLAUDE.md を更新
5. 次月のパフォーマンスを計測・フィードバック
```

**制約**: `agents/*/CLAUDE.md` の最終更新権限は**社長（人間）**が保持する。
エージェントは「提案」まで、「実行」は社長承認後のみ。

### Phase 4: 完全自律運用（理想状態）

| 機能 | 説明 |
|---|---|
| 自己スケジューリング | 相場状況に応じて監視頻度を動的に調整 |
| 自己修復 | Azure Functions の障害を検知して再起動・通知 |
| 自己学習 | 過去の投資判断データから判断ロジックを継続改善 |
| 自己報告 | 週次でパフォーマンスサマリーを社長に自動送信 |

### 自律改善を機能させる前提条件

1. **観測可能性（Observability）**: 全エージェントの判断・行動がログに残ること
2. **フィードバックループ**: 判断結果の正誤が後から分かる仕組み（投資結果の記録）
3. **提案と承認の分離**: エージェントは提案のみ、実行は人間が承認
4. **段階的委譲**: Phase 1→2→3→4 の順で、実績を積んでから次のフェーズへ

## 7. Harness Engineering 実践（記事5：HOW MUCH）

スタートアップが21体のAIエージェントで開発を回すための「ハーネス」設計知見。

### 7.1 プロセス分解の徹底
要件 → 仕様 → 基本設計 → 詳細設計 → 実装 → タスク分割 → featureブランチ → devマージ。
各段階の粒度を細かく区切ることで、AIが確実に実行できるチェックポイントを作る（モノリシックなタスクは失敗する）。

### 7.2 GitHub完結原則
**「開発のすべてをGitHub上で完結させる」**。
ローカルでのブラックボックス開発を排除し、AIの行動を全てPR・Issue・ラベル・Actionsログとしてトレース可能にする。

### 7.3 ラベル駆動オーケストレーション
バトンパス自動化：あるエージェントが作業完了時にラベルを付与すると、次のエージェントが起動する。
人間の手動介入を排除する仕組みとして強力。

### 7.4 アーキテクチャ制約のコード化
プロンプトベースの「Golden Principles」 → カスタムlinterへ進化させる。
依存層違反など、機械的に検出可能なルールはlinter化して伝播を防ぐ。

### 7.5 多段CI/CD
- **破壊的スキーマ変更検出**: AI生成マイグレーションによるデータ損失防止
- **monorepo横断 lint/test**
- **ドキュメント同期検証**: コードとdocsの乖離を検出

### 7.6 停止基準・エスカレーション
- 明示的な反復上限（例: `review cycle max: 5`）
- 上限到達時は **「人間の対応が必要」ラベル** を付与してエスカレーション
- 無限ループ防止の最後の砦

### 7.7 開発リズム（奇数日／偶数日）
- 奇数日: 機能開発
- 偶数日: パイプライン（ハーネス）改善
- インフラ変更が実際に機能するか、機能日で即検証する設計

### 7.8 スコアリング駆動リファクタリング
6軸（アーキ・コード品質・テスト・セキュリティ・性能・運用性）でリポジトリの健康度を定期評価し、低スコア領域に改善Issueを自動生成。

### 7.9 セルフヒーリング監視
5分ごとに「詰まったパイプライン」を検出し、安全な範囲で自動回復（ラベル再付与・workflow再実行）。失敗時のみ人間にエスカレーション。

### 7.10 プレビュー環境の自動化
Terraformで PRごとに独立DNS・QRコード付き環境を自動構築 → PRクローズで自動破棄。

### 本リポジトリへの適用案
- **CIO判定リトライ上限の導入**: 投資判断パイプラインで明示的な停止基準を設定
- **ラベル駆動バトンパス**: `agents/development/` のスプリント実行に GitHub Issue ラベルベースの引き継ぎを検討
- **月次ヘルススコアリング**: CoSが6軸で本リポジトリを採点し改善Issueを自動起票
- **破壊的スキーマ変更検出**: Cosmos DB スキーマ変更を CI で検出（投資データ損失防止）
- **ハーネス改善デー**: スプリントの偶数日相当を `.claude/` 配下の改善に充てる

## 8. AIエージェント実装パターン（記事6：Google ADK）

エージェントを「アプリケーション」として実装するときの設計パターン。

### 8.1 イベントストリーミング（SSE）
REST API ではなく **SSE で型付きイベント** をpush。
- `Run started` / `Text delta` / `Tool call` / `Tool result` / `Run finished`
- フロントエンドはインクリメンタルレンダリングで進捗を可視化

### 8.2 サーバーサイド会話管理
セッションサービスで会話履歴・状態を一元管理。フロントエンド同期が単純化する。

### 8.3 状態スコープの分離
| スコープ | 用途 |
|---|---|
| session | 単一会話のみ有効 |
| user | 特定ユーザーで永続 |
| app | アプリ全体で共有 |
| temp | 一時的（ツール呼び出し中など） |

### 8.4 プロンプトでのツール使用は具体的に
- ❌「必要に応じて使用してください」
- ✅「ユーザーが 'today' と言ったら必ず `current_time` を呼び出すこと」

曖昧な指示はツール呼び出しの取りこぼしを生む。

### 8.5 ダイナミックプロンプト
インストラクション内で `{user:preferred_language}` のような状態参照を使い、実行時に置換する。

### 8.6 マルチエージェント組織
- **転送 vs ツール化の使い分け**:
  - サブエージェントがユーザーに直接応答すべき → **転送**パターン
  - 結果を親が処理して返す → **ツール化**パターン
- **SequentialAgent**: 多段パイプライン化、各段の結果を次段に渡す
- **関心の分離**: 各サブエージェントを狭い職務に特化、親がオーケストレーションのみ担当

### 8.7 グラウンディング戦略
LLMは訓練データ以降の情報を知らない。常に以下を提供：
- **GoogleSearch**（最新Web情報）
- **current_time**（現在日時）
- **URLContext**: ユーザー参照URLを直接取得させ、推測を排除

### 8.8 MCP統合
`mcptoolset` で外部プロバイダーを標準化。エージェント間で機能共有を促進。

### 本リポジトリへの適用案
- **Researcherエージェントの強化**: 「URL言及時はWebFetch必須・推測禁止」を `agents/investment/research/CLAUDE.md` に明文化
- **投資判断時の必須ツール明示**: 判断時は必ず `current_time` ＋ 最新ニュース取得をツール呼び出し必須にする（時刻トリガー修正の発展形）
- **Discord/Slack通知のインクリメンタル化**: 長時間タスクの進捗をSSE風に逐次送信
- **状態スコープ分離**: 投資判断結果は `app` スコープ、ユーザー設定は `user` スコープと明示的に区別
- **SequentialAgent的設計**: Quant → Researcher → RiskManager → CIO のパイプライン化を CoS が機械的に実行
