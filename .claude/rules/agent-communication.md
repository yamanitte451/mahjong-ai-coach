# エージェント間コミュニケーション規約

## メッセージング方針
- チームメイト同士は**直接メッセージ**でやり取りできる
- **ブロードキャスト**（全員宛）はコストが高いため、本当に全員に必要な情報のみ使用
- 部門内の連携は積極的に行う

## メッセージフォーマット
```
【送信元】{エージェント名}
【宛先】{エージェント名 or ALL}
【種別】報告 / 依頼 / 質問 / アラート
【内容】
{本文}
```

## 典型的な連携パターン

### 事業検討フロー
```
1. CoS → CSO, CFO を並列スポーン
2. CSO → CFO: 市場分析結果を共有（財務試算の前提として）
3. CFO → CoS: 市場分析を踏まえた収益試算を報告
4. CoS → CMO, CTO, CLO を必要に応じて追加スポーン
5. CoS → 社長: 統合レポートを提示
```

### 開発フロー
```
1. CoS → PM をスポーン（仕様確認・feature slice 分解）
2. PM → QA: 受け入れ基準とテスト観点を事前共有
3. PM → Backend, Infra: 受け入れ基準・非ゴール・既知リスク付きで実装依頼
4. Backend, Infra: 失敗するテスト / 検証シナリオを先に用意し、その後に最小実装
5. Backend, Infra → QA: 実装完了時に自動でレビュー依頼を送信
6. QA → Backend, Infra: 不具合やテスト漏れがあれば差し戻し
7. QA → PM: テスト結果を報告
8. PM → CoS: QA結果込みのスプリント状況を報告
```

## 仕様・受け入れ基準の共有ルール

- PM / CoS は、実装依頼の前に **参照した元仕様ファイル** を明示する
- 実装依頼には、最低限 **対象機能 / 非ゴール / 受け入れ基準 / 既知リスク** を含める
- 受け入れ基準は `Given / When / Then` または同等の検証可能な表現にする
- QA は「テストがあるか」だけでなく、**受け入れ基準に未対応の観点がないか** を確認する

## 自動 QA レビュー規則

- コード、テスト、インフラ、CI/CD、DB スキーマ、実行時設定、またはエージェント挙動に影響する指示ファイルの変更時は、**QA レビューを必須ゲート**とする
- PM / CoS は、ユーザーの追加指示がなくても **実装担当の完了報告をトリガーに** QA へ自動でバトンを渡す
- docs-only の変更は原則スキップ可。ただし、**開発フロー・CI・レビュー規約・エージェント挙動**を変える文書変更は QA 対象
- QA へ渡す内容は、変更ファイル、要件 / 受け入れ基準、実行済みチェック結果、既知リスク、未解決事項のみに絞る
- **blocking** とは、QA が修正必須と判定した指摘が 1 件でも残っている状態を指す。blocking が残る限り完了報告しない
- QA 指摘があれば実装担当へ差し戻し、修正後に再レビューする
- 修正 -> 再 QA のサイクルは最大 3 回まで。超えたら **ユーザー（社長）** 判断へエスカレーションする

## ファイル競合回避ルール

| ファイル | 所有者 | 備考 |
|---|---|---|
| business_context.md | CoS（リーダー） | 全員参照可・更新はCoSのみ |
| agents/*/CLAUDE.md | 読み取り専用 | ロール定義・編集不可 |
| src/ 配下のコード | Backend / Infra | 担当ファイルを分割して作業 |
| tests/ 配下 | QA | テストコードはQAが担当 |

## 統合レポート形式

CoS が社長に提示する最終レポートは以下の形式：

```
## 📋 会議サマリー
### 参加エージェント
[スポーンしたエージェント一覧]
### 各部門からの報告
[各エージェントからのメッセージを統合]
### 意見の相違点
[エージェント間で見解が分かれた点を明示]
### 推奨アクション
[優先度付きのネクストステップ]
### 社長への確認事項
[意思決定が必要な項目]
```
