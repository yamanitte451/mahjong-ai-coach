# コンテキスト管理ルール（トークン節約）

## エージェントスポーン時のコンテキスト最小化

### 悪い例（トークン浪費）
```
business_context.md, CLAUDE.md, src/xxx.py, ... をすべて読んでください
```

### 良い例（必要な情報のみ）
```
【タスク】○○機能の実装
【参照仕様】docs/spec.md の 2.3 ○○画面
【受け入れ基準】
- Given: 必要なデータが登録済み
- When: 対象の画面を開く
- Then: 期待する情報が表示される
【非ゴール】△△機能は今回含めない
【既知リスク】○○の集計ロジック
【対象ファイル】src/xxx.py（参考実装: src/yyy.py）
```

### 良い例（自動 QA レビュー）
```
【タスク】○○機能の QA レビュー
【変更ファイル】src/xxx.py, tests/test_xxx.py
【要件 / 受け入れ基準】A の入力で B になること
【実行済みチェック結果】unit: pass, integration: not run
【既知リスク】境界値の手動確認が未実施
【未解決事項】例外系 UX の扱いを未確定
```

### 良い例（TDD の最小ハンドオフ）
```
【タスク】○○機能の実装
【参照仕様】docs/spec.md の 2.5 ○○セクション
【受け入れ基準】
- Given: 必要なデータが存在する
- When: 対象の処理を実行する
- Then: 期待する結果が返される
【先に用意するテスト】
- 正常系の unit test
- データなしケースの empty-state test
```

## 階層型オーケストレーション原則

1. CoS（自分）がプラン確定 → 必要な情報を抽出してワーカーへ渡す
2. ワーカーには「自分のロールファイル + タスクに直接必要なファイル」だけ
3. ワーカーの出力はサマリーのみ受け取り、全文をコンテキストに積まない
4. 自動 QA レビュー時も、変更差分すべてではなく **変更ファイル、要件 / 受け入れ基準、実行済みチェック結果、既知リスク、未解決事項** だけを渡す
5. 実装依頼時は、差分全文ではなく **参照仕様・非ゴール・受け入れ基準・先に書くテスト** を優先して渡す
