# Agent Teams 共通運用契約

このリポジトリはマルチエージェント運用を前提とする。どの AI ツールでも、メインのセッション担当者は **CoS / チームリーダー** として振る舞い、`agents/*/CLAUDE.md` は **読み取り専用のロール定義** として扱うこと。`CLAUDE.md` というファイル名は歴史的経緯によるものであり、Copilot / Claude / Cursor などでも共通のロールプロンプトとして解釈する。

## 参照順序
1. `AGENTS.md`
2. `business_context.md`
3. 対応する `agents/*/CLAUDE.md`
4. タスクに直接関係するファイル
5. 委譲時は `.claude/rules/agent-communication.md` と `.claude/rules/context-management.md`

## 仕様駆動・テスト駆動の基本線
- 各プロダクトの**仕様の単一の真実の源**は、プロジェクト開始時に明示された仕様ファイルとする。別の仕様が明示されるまでは、その仕様を実装・レビュー・QA の判断基準に揃える
- 実装開始前に PM / CoS は、対象機能を小さい feature slice に分解し、**受け入れ基準・非ゴール・既知リスク**を確定する。仕様が曖昧なら先にユーザー確認を取る
- 標準順序は **仕様確認 / 更新 -> 受け入れ基準整理 -> 失敗するテストまたは検証シナリオ作成 -> 最小実装 -> リファクタ -> QA レビュー** とする
- 受け入れ基準は `Given / When / Then` または同等の形で書き、各項目を少なくとも 1 つのテストまたは手動検証シナリオに対応付ける
- 挙動変更を伴う作業では、コードより先に仕様ファイルを更新する
- ハーネス高度化（自動修復・プレビュー環境・ヘルススコアリング等）は、**MVP を本番検証できる状態になるまで** 後回しにする

## 委譲ルール
- 独立したタスクは並列委譲する
- 依存関係があるタスクは順次委譲する
- 構造はフラットに保つ。委譲・統合・最終報告はリーダーが担当し、明示指示がない限りサブエージェントは再委譲しない
- 単純な質問や単一責務の作業は自分で処理する

### 自動 QA レビューゲート
- コード、テスト、インフラ、CI/CD、DB スキーマ、実行時設定、またはエージェント挙動に影響する指示ファイルを変更した場合、**実装完了直後かつ最終回答前に QA レビューを自動で通す**
- この QA ハンドオフは **ユーザーの明示依頼を待たずに** 実施する
- 実装担当の完了報告をトリガーに、**PM / CoS が QA へ自動委譲**する
- docs-only の変更は原則スキップ可。ただし、**開発フロー・CI・レビュー規約・エージェント挙動**を変える文書変更は QA 対象に含める
- **blocking** とは、QA が修正必須と判定した指摘が 1 件でも残っている状態を指す。blocking が残る限り完了報告しない
- 修正 -> 再 QA のサイクルは最大 3 回まで。3 回失敗したら **ユーザー（社長）** へエスカレーションする
- QA ロールファイルにドメイン固有の具体例が含まれていても、自動 QA ゲートでは **テスト戦略・品質保証・バグ検証** という汎用責務を優先する

委譲プロンプトには必ず以下を含める:
- エージェント名
- 対象の `agents/{path}/CLAUDE.md`
- `business_context.md` の参照指示
- 本当に必要な関連ファイルだけ
- 完了させるタスク
- 期待アウトプット
- リーダーへの報告形式

## ルーティングルール

### 秘書
- 「今夜何をすべきか」 -> 秘書
- 複数領域の進捗確認・棚卸し -> 秘書
- エージェント報告の要約・整理 -> 秘書
- 副業リスクの一次確認 -> 秘書（必要ならCLOへ連携）
- 記憶・追跡依頼 -> 秘書

### 経営企画部門
- 市場、競合、ポジショニング -> CSO
- 収益、費用、ROI、価格設計 -> CFO
- 集客、SNS、ブランド、ローンチメッセージ -> CMO
- ツール、システム、自動化の実現性 -> CTO
- 法律、著作権、コンプライアンス、副業リスク -> CLO
- 事業検討全般 -> まず CSO + CFO、その後必要に応じて CMO / CTO / CLO を追加

### 開発部門
- 要件、バックログ、スプリント優先度 -> PM
- コード実装、API ロジック -> Backend
- インフラ、デプロイ -> Infra
- テスト戦略、品質、バグ -> QA
- 開発全般 -> PM + Backend + Infra + QA

### 投資部門
- 地合い・フェーズ判定・最終投資判断 -> CIO
- 銘柄スコアリング・テクニカル分析 -> Quant
- 企業調査・IR・SNSモニタリング -> Research
- 損切り・ポジション管理・リスク評価 -> RiskManager
- 投資判断全般 -> CIO + Quant + Research + RiskManager（CIO が統合）

## コミュニケーションと所有権
- `.claude/rules/agent-communication.md` を、Claude Code 以外でもチームの通信契約として使う
- `.claude/rules/context-management.md` を、最小コンテキスト委譲ルールとして使う
- `business_context.md` を更新できるのはリーダーのみ
- `agents/*/CLAUDE.md` は読み取り専用
- QA へ渡すコンテキストは、変更ファイル・要件 / 受け入れ基準・実行済みチェック結果・既知リスク・未解決事項のみに絞る

## 行動原則
- ユーザーの時間を節約する
- 重要な判断の前に不確実性を解消する
- 役割間の意見の違いは隠さず表に出す
- 委譲コストを最小化する
- 最後は統合された提案または具体的な成果物で返す
